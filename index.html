Starting Off

Jul 23rd, 2013

So well, I’m trying (yet again) to sit down and write and express things through a nifty blogging mechanism — Octopress. Why I chose this over Wordpress?

  • Hmm, I think it has a bit to do with the fact that this was somewhat harder to setup — hence, the novelty factor.
  • As an author I can control far more things, far easier. Like the placing of text blocks and images.
  • Code blocks — oh they render so well. I know there’s prettify and what not on WP, but the fact remains, it’s all baked into Octo already.
    Case in point —
  • Writing and formatting stuff in markdown is way more natural than in WP.
  • No database required!
  • The installation and deployment process takes one through the narrow and often winding lanes of github (i.e, if you chose to host it on pages, ruby, git, saas, yaml, and some more which I’ve ignored right now.
  • and of course, it’s got this whole geeky feel to it. There’s this sense of accomplishment associated with the whole process.
  • free deployment against your own custom URL using Github pages. Last I saw, WP was charging for that!

It’s not all easy-peasy though!

From the point you decide to have an Octo blog, to finally having one deployed can be quite a handful. Especially more so, if you got a bit self assured of your own obvious intelligence. Well I did.
I was using a Mac, and the first thing is to get Ruby 1.9.3 installed. It was no fun, since I apparently had Ruby 1.8.x preinstalled, and didn’t have much clue how to go about uninstalling it. Suggestions on the web generally indicated that all you need to do is change the PATH variable. But then, I ran into some shit storm since I didn’t have gcc installed for compiling the binaries. This is because I recently updated my XCode, and that took away the command lines tools installed before. So before I could figure that out, I had gotten into MacPorts, Homebrew and Fink. They all were basically puking into each others dependencies. And it all had to undone.

First thing to make sure — have you got your command line tools installed
on XCode?

Once the gcc is taken care of, the next thing I got entangled with was Github pages. This part, I believe, had more to do with my impatience than any real issue as such. You need to be a little careful about the documentation and how the folder structure of your Octo clone maps to Github pages. When you do a git clone on the original Octo repo, you get what is known as the root directory. When you run the standard Octo rake commands, like generate, deploy, etc, another directory _deploy is created within the source directory. This _deploy directory is also a valid git project.
Now, on the Github pages side of things, depending on what kind of blog you want to float – personal, official, team – you’ll need to create a new repository for that. Now, the master branch should map to the _deploy folder, while a new branch source should map to the root folder. If all this sounds a bit hard to follow, just be sure that you fire the rake commands in the correct order as instructed in the documentation.

Don’t make the mistake of fiddling around with the root folder setting in the
_config.yml file, or through the rake set_root_dir[] command before getting
your blog live!

If you do something like that, chances are, your JS, CSS and other linked files will not get resolved properly. In case you’ve done anything like that, make sure that the root setting under Jekyll parameters in the _config.yml file is set to empty. And then, execute the commant rake set_root_dir[/] to fix the relative paths generated in the generated index.html file.

Getting to See the Damn Thing

You can view the blog locally before pushing it live. To do so, use the rake generate and then rake preview command. Then access your blog at localhost:4000. You can even edit your posts in realtime and review the changes.
To deploy your blog live on Github pages, use the command rake deploy (assuming you have already run rake generate before). This will create all the new files in the _deploy folder and push these files to the master branch on your Github pages repository. All you need to do then is access your live blog at http://<github-username>.github.com (the .com or .io is the same thing. No difference.)
